# Phase 210 — init path decision contract

## At a glance

| Field | Value |
|---|---|
| Phase family | Phase 2 — bootable image |
| Run command | `make phase 210` |
| Underlying make target/script | `vm/phase2/verify-init-decision.sh` |
| Runs on | host |
| Main proof/artifact | Records the current init decision: systemd-on-musl plus systemd-boot. |


Phase 210 turns the Phase 209 probe into an explicit project decision.

Phase 210 does not build the init system.
It does not install systemd.
It does not mount the image.
It does not boot QEMU.

It only records how ONIX will proceed.

## Why a decision needs its own phase

Phase 209 produced *evidence* ("a musl systemd derivation exists and is
plausible"). Evidence is not the same as a commitment. Phase 210 exists to write
that commitment down in a form the tooling can *check* — a contract — so that
every later phase, and every future contributor, builds against the same
foundational choice instead of quietly re-litigating it.

This is the ONIX habit of making decisions **explicit and verifiable** rather
than implicit and remembered. The phase is a gate not because it does work, but
because it says: *from here on, the image is a systemd machine.* If someone later
changed the boot skeleton to a different init path, this phase's check would
fail and catch the drift.

The verifier is `vm/phase2/verify-init-decision.sh`, and it is deliberately
humble. It does two things, both host-only:

1. Greps this page (`book/src/phases/210.md`) for the exact decision statements
   listed below (`init path: systemd-on-musl`, `bootloader: systemd-boot`,
   `ONIX uses systemd as PID 1`, and so on).
2. Greps the image-build script `vm/phase2/build-image-skeleton.sh` to confirm
   the boot skeleton it produces still carries `init=/usr/lib/systemd/systemd`
   and still describes itself as a `systemd-boot/BLS skeleton`.

That second check is the important one: it ties the *written* decision to the
*actual* code that stamps the boot entry, so the words and the machine cannot
disagree.

#### The decision

The Phase 210 decision is:

```text
init path: systemd-on-musl
bootloader: systemd-boot
```

Project rule:

```text
keep systemd if we can
```

In plain words: ONIX uses systemd as PID 1.

That means we continue with the systemd path for now because Phase 209 showed:

```text
pkgsMusl.systemd exists
pkgsMusl.systemd targets musl
pkgsMusl.systemd is not marked broken
pkgsMusl.systemd uses -Dlibc=musl
Nix can plan the build graph
```

But we do not pretend that systemd is proven boot-ready. The systemd path still
has to prove itself in real ONIX builds and boots.

#### Why bootloader and init are separate

The bootloader chooses and launches the kernel.

The init system is the first userspace process after the kernel mounts the real
root filesystem.

Those are different jobs:

```text
systemd-boot  -> bootloader
systemd       -> init system / PID 1
```

They happen at different moments and in different worlds. The **bootloader** runs
in firmware-land, before Linux exists: after the UEFI firmware (OVMF, in QEMU)
hands off, systemd-boot presents boot entries, picks one, loads the kernel image
and initramfs into memory, and jumps into the kernel. Its job is over the instant
the kernel starts. The **init system** runs in Linux-land: it is the very first
userspace process the running kernel executes, and it stays alive for the entire
uptime of the machine. Confusingly, both ship in the systemd project and both
have "systemd" in the name — but they are separate programs doing separate jobs.
ONIX happens to choose the systemd option for *both* slots, which is why the
decision has to spell out which "systemd" it means where.

That is why the decision names both parts explicitly:

```text
systemd-boot loads the kernel
systemd runs as PID 1
```

#### Background: BLS and the kernel command line

systemd-boot does not hard-code its menu. It reads **BLS** (Boot Loader
Specification) entries — small text files, one per bootable configuration, that
name a kernel, an initramfs, and a *kernel command line*. ONIX names these
entries `onix-<txid>.conf` (during Phase 2 they are simpler, e.g.
`onix-phase-211.conf`). The **kernel command line** is a string of options the
bootloader passes into the kernel; two options on it encode this phase's
decision directly, which is what the next section is about.

#### What this means for the boot entry

The BLS entry keeps this kernel command line intent:

```text
init=/usr/lib/systemd/systemd systemd.unit=multi-user.target
```

Read those two options as the decision made executable:

- `init=/usr/lib/systemd/systemd` tells the kernel *which program to run as PID
  1*. By default the kernel runs `/sbin/init`; ONIX overrides that to point
  straight at the systemd binary. This is the literal handoff address.
- `systemd.unit=multi-user.target` tells that systemd, once running, *how far to
  boot* — to the multi-user (non-graphical, networked) target rather than, say, a
  rescue shell.

That path is the userspace handoff point:

```text
kernel -> root filesystem -> /usr/lib/systemd/systemd
```

So Phase 211 and later must place a real systemd userspace at that path.
Right now nothing lives there yet — the decision is a *promise* the image has to
make good on. Phase 211 puts a kernel and initramfs in place so there is
something to hand *from*; Phase 213 is the phase that finally installs a real
musl systemd at `/usr/lib/systemd/systemd` so there is something to hand *to*. If
that path is empty when the kernel reaches it, the boot dies with a panic — which
is exactly the failure Phase 212's boot probe is built to observe.

#### What Phase 210 verifies

`make phase 210` verifies:

- this Phase 210 section exists
- the init path is `systemd-on-musl`
- the bootloader is `systemd-boot`
- the project rule says `keep systemd if we can`
- the plan says ONIX uses systemd as PID 1
- the boot entry points at `/usr/lib/systemd/systemd`
- the plan mentions `systemd starts as PID 1`
- the plan mentions `udev/device setup works`
- the plan mentions `basic services work`
- the Phase 206 boot skeleton still follows the systemd path

#### What Phase 210 does not prove

Phase 210 does not prove:

```text
systemd builds
systemd boots
services work
QEMU reaches login
```

It only makes the next engineering decision explicit.
