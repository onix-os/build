# Phase 002 — build the forge disk

## At a glance

| Field | Value |
|---|---|
| Phase family | Phase 0 — forge |
| Run command | `make phase 002` |
| Underlying make target/script | `vm/phase0/fetch-rootfs.sh + vm/phase0/build-disk.sh` |
| Runs on | host, with rootful loop/mount/chroot work |
| Main proof/artifact | Creates the bootable forge disk at vm/state/quarry.raw. |


```sh
make phase 002
```

This creates the bootable forge disk:

```text
vm/downloads/alpine-minirootfs-*.tar.gz
        |
        v
vm/state/quarry.raw
```

The disk layout is intentionally simple:

```text
ONIX-ESP    FAT32 EFI system partition
onix-root   ext4 root filesystem
```

## What this step turns into what

The input is a 3.7 MB Alpine *minirootfs* tarball; the output is a 20 GiB
bootable UEFI disk image at `vm/state/quarry.raw` with a musl toolchain, an SSH
server, and a bootloader already installed. Two scripts do this:

- **`fetch-rootfs.sh`** downloads the pinned Alpine minirootfs and verifies its
  SHA-256 against `config.sh` (idempotent — it skips the download if the file is
  already present and matches). The pin is `alpine-minirootfs-3.24.1-x86_64`
  with a recorded checksum, so the seed is reproducible; if Alpine bumps the
  point release the script fails loudly rather than silently using new bytes.
- **`build-disk.sh`** does the privileged disk assembly, and hands the actual
  in-image setup to **`chroot-setup.sh`**.

> **Sidebar — what a minirootfs is.** A *rootfs* (root filesystem) is the
> complete directory tree a Linux system boots into: `/bin`, `/etc`, `/lib`,
> and so on. Alpine's *minirootfs* is the smallest useful one — busybox, musl,
> `apk` (Alpine's package manager), and little else, about 3.7 MB compressed.
> It is not bootable on its own (no kernel, no bootloader), but it is a perfect
> seed: unpack it, `chroot` in, and use `apk` to add exactly the packages you
> need. We use the tarball, **not** Alpine's ISO installer, because we want to
> control every byte that goes on the disk.

## Why UEFI, GPT, FAT32 ESP, and ext4 root

The disk uses a **GPT** partition table with two partitions, mirroring how real
UEFI machines boot:

> **Sidebar — GPT and the ESP.** Modern PCs boot via **UEFI** firmware, which
> reads partitions from a **GPT** (GUID Partition Table). UEFI firmware can only
> read one filesystem: **FAT**. So every UEFI disk has an **ESP** (EFI System
> Partition), a small FAT partition where bootloaders live as `.EFI` files. The
> firmware loads `\EFI\BOOT\BOOTX64.EFI` (the "removable media" fallback path)
> automatically. That is why the ESP here is **FAT32** and labeled `ONIX-ESP` —
> it has to be FAT for the firmware, and the removable path lets a plain QEMU
> boot with no NVRAM boot entry configured.

The root partition is **ext4**, labeled `onix-root`. ext4 is a journaled Linux
filesystem — the forge does not need anything fancier. (Note this is the *forge*
disk. The real ONIX image targets XFS for root and adds XBOOTLDR + persist
partitions; the forge is deliberately the simpler two-partition shape.)

Labels matter: `chroot-setup.sh` writes an `/etc/fstab` that mounts by
**label** (`LABEL=onix-root`, `LABEL=ONIX-ESP`), and GRUB's config searches for
the root by label too. Label-based mounting survives the device name changing
between build-time (`/dev/loop7p2`) and boot-time (`/dev/vda2`).

## Walk the build

During the build, the script:

1. creates a sparse raw disk
2. attaches it with `losetup`
3. partitions it
4. formats the ESP and root partition
5. extracts Alpine minirootfs
6. enters a chroot
7. installs kernel, OpenRC, SSH, GRUB, Rust/build tools
8. creates the `mason` build user
9. exports kernel/initramfs for direct boot fallback

Expanded, so you can see what each step actually does:

1. **Sparse raw disk.** `truncate -s 20G quarry.raw` creates a file that
   *claims* 20 GiB but occupies almost no real disk until data is written to it
   (a "sparse" file). Room for Rust builds and `.stone` output without paying
   for 20 GiB up front.
2. **Attach with `losetup`.** `losetup --find --show --partscan` maps the file
   to a free `/dev/loopN` and, thanks to `--partscan`, exposes its partitions
   as `/dev/loopNp1`, `/dev/loopNp2` once they exist. The script waits (up to a
   few seconds, re-running `partprobe`) for those partition nodes to appear.
3. **Partition (GPT).** `sgdisk --zap-all` wipes any old table, then two
   `sgdisk -n` calls create partition 1 (`+512M`, type `EF00` = ESP, name
   `ONIX-ESP`) and partition 2 (rest of disk, type `8300` = Linux, name
   `onix-root`).
4. **Format.** `mkfs.fat -F32 -n ONIX-ESP` on p1 and `mkfs.ext4 -L onix-root`
   on p2 lay down the two filesystems with those volume labels.
5. **Extract minirootfs.** Both filesystems are mounted (root at a temp mount,
   ESP under `…/efi`), then `tar -xpf` unpacks the Alpine tarball into root.
   The `-p` preserves permissions. Alpine's `apk` repositories and a DNS
   `nameserver` are seeded into `/etc` so the chroot can reach the network.
6. **Enter a chroot.** The script mounts an *isolated* `/proc`, `/sys`, and a
   fresh `tmpfs` `/dev` inside the target, creates the handful of device nodes
   (`null`, `zero`, `random`, …) the tools need, and exposes only this build's
   loop block nodes (so GRUB probes the right disk, not the host's). Then it
   runs `chroot "$MNT" /bin/sh -e /root/chroot-setup.sh`.

   > **Sidebar — chroot.** `chroot` runs a process with a *different directory
   > tree as its `/`*. Inside the chroot, `/bin/sh` is the new rootfs's shell,
   > `apk` installs into the new rootfs, and paths like `/boot` refer to the
   > image, not the host. It is how you configure a system image "from the
   > inside" without booting it — but it shares the host kernel, so `/proc`,
   > `/sys`, and `/dev` must be mounted in for tools to work.

7. **Install kernel, OpenRC, SSH, GRUB, Rust/build tools** (this is
   `chroot-setup.sh`'s job — see below).
8. **Create the `mason` build user.** A non-root user (default name `mason`,
   from `config.sh`) that will run boulder and moss. It gets a password, wheel
   group + `doas` for sudo-like escalation, the host's generated SSH public key
   in `authorized_keys`, and — importantly — `subuid`/`subgid` ranges.

   > **Sidebar — why subuid/subgid.** boulder builds each package inside an
   > unprivileged Linux **user namespace**, so a non-root user can create an
   > isolated sandbox with its own "root." That requires a range of subordinate
   > UIDs/GIDs mapped to the user, declared in `/etc/subuid` and `/etc/subgid`.
   > Without them, boulder's rootless sandbox cannot set up its build root. The
   > line written is `mason:100000:65536`.

9. **Export kernel + initramfs.** The built kernel (`vmlinuz-virt`) and its
   generated `initramfs-virt` are copied out to `vm/state/` so `launch.sh
   --direct` can boot the kernel directly, bypassing GRUB entirely. This is a
   safety net for when the bootloader misbehaves.

### What `chroot-setup.sh` installs and why

Running inside the new rootfs, it `apk add`s: the Alpine base + `linux-virt`
kernel + `mkinitfs`, **OpenRC** + busybox-openrc (init/service manager), **GRUB**
+ `grub-efi` + `efibootmgr` (bootloader), **OpenSSH** + `doas` + `shadow`,
filesystem tools (`e2fsprogs`, `dosfstools`, `xfsprogs`), CA certificates, and
the whole build toolchain moss/boulder needs: `bash build-base git just rust
cargo clang llvm-dev binutils cpio libarchive-tools musl-dev linux-headers
pkgconf openssl-dev zlib-dev xz-dev`.

It then sets the hostname to `quarry`, writes the label-based `/etc/fstab`,
regenerates the initramfs with virtio/ext4/xfs drivers (so the VM can mount both
the forge disk and later ONIX disks), installs GRUB to the ESP in `--removable`
mode (writing `\EFI\BOOT\BOOTX64.EFI` and a `grub.cfg` with a serial console),
creates the `mason` user, hardens sshd to key-only login, enables DHCP on
`eth0`, adds a serial getty on `ttyS0`, and registers the OpenRC services
(`sshd`, `networking`, `crond`, etc.). `build-disk.sh` then double-checks the
removable EFI loader really landed on the ESP, and rebuilds a standalone GRUB
image if GRUB wrote it to the wrong place.

## Why OpenRC and GRUB here are throwaway

OpenRC and GRUB are only forge scaffolding. The real ONIX target can still
use systemd/systemd-boot later if we choose that route.

This is a deliberate and important point. The forge uses whatever Alpine makes
easy — OpenRC as init, GRUB as bootloader — because the forge's only job is to
compile moss+boulder and cut stones. **None of it ships in ONIX.** The real ONIX
image (Phase 2 onward) targets **systemd as PID 1** with **systemd-boot / BLS
Type #1** boot entries (`onix-<txid>.conf`), not OpenRC and GRUB. Using GRUB in
the forge is not a commitment; it is just the fastest way to get a bootable
Alpine VM.

> **Sidebar — init and bootloader, quickly.** The *bootloader* (GRUB here) is
> the firmware-launched program that loads the kernel and initramfs. *PID 1* /
> *init* (OpenRC here) is the first userspace process the kernel starts; it
> mounts filesystems and starts services. ONIX will replace both — systemd-boot
> for the loader, systemd for PID 1 — but the forge does not need ONIX's
> machinery to do its job.

## What it proves — and what it does not

**Proves:** we can turn a bare musl minirootfs into a bootable, SSH-able,
toolchain-equipped disk image, fully unattended, reproducibly from a pinned
seed.

**Does not prove:** that it boots (that's [003](./003.md)), or that moss/boulder
build on it (that's [004](./004.md)). It only proves the disk was *assembled*
correctly — the build script verifies the EFI loader exists on the ESP before
declaring success.

## Gotchas and how to read them

- **Stale loop mounts.** If a previous build was interrupted, loop devices and
  temp mounts can linger. `build-disk.sh` refuses to run if it detects stale
  ONIX mounts and tells you to run `make cleanup`; there is also a
  `--cleanup-stale` mode that detaches them.
- **No OVMF firmware.** The build wants UEFI firmware available; if
  `ONIX_OVMF_CODE`/`ONIX_OVMF_VARS_TEMPLATE` are unset and none is found, it
  dies with a clear message (the Nix flake normally exports these).
- **Rebuilding.** The disk is not overwritten if it already exists; pass
  `--force` (or `make` re-run after `./clean.sh`) to rebuild.
- **Inspecting.** `./build-disk.sh --inspect` mounts the finished image
  read-only and lists the ESP and `/boot` contents — useful for confirming the
  loader and kernel landed where expected.
