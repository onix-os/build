# Phase 004 — provision tools

## At a glance

| Field | Value |
|---|---|
| Phase family | Phase 0 — forge |
| Run command | `make phase 004` |
| Underlying make target/script | `vm/phase0/ssh.sh mason ./provision.sh` |
| Runs on | guest over SSH |
| Main proof/artifact | Builds moss and boulder inside the forge. |


```sh
make phase 004
```

This SSHes into the running forge and builds:

- `moss`
- `boulder`

from the pinned `os-tools` commit in `config.sh`.

## Why this is the heart of Phase 0

Everything so far — download, disk, boot — existed to reach *this* moment: a
musl machine that can compile the ONIX tooling. Phase 004 is where we actually
get moss and boulder, the two programs the entire distribution is built on.

> **Sidebar — moss and boulder.** Both are Rust binaries from AerynOS's
> **`os-tools`** repository, the *one external dependency* ONIX takes. They do
> two complementary jobs:
> - **boulder** is the **builder**. Given a recipe (`stone.yaml`), it fetches
>   the sources, runs the build in a sandboxed user namespace, and emits a
>   `.stone` package.
> - **moss** is the **package/state manager**. It installs `.stone`s into a
>   root, tracks every change as an atomic *state*, and can roll back to any
>   previous state.
>
> ONIX borrows AerynOS's *tooling* (moss, boulder, the `.stone` format) but
> **none of its packages** — AerynOS is glibc; ONIX is musl. Building these two
> tools *on musl* is the proof that a musl-managed machine plane is even
> possible.

## What `provision.sh` actually does

`make phase 004` runs `vm/phase0/ssh.sh mason ./provision.sh` — it logs into the
running forge as `mason` (key-only) and executes the `provision.sh` that was
baked into the image in Phase 002. Inside the guest it:

1. **Prints toolchain versions** (`rustc`, `cargo`, `just`) to confirm the build
   environment `chroot-setup.sh` installed is present.
2. **Clones or updates `os-tools`** into `~/src/os-tools` and checks out the
   pinned commit `OS_TOOLS_REF` (from `config.sh`, also written to
   `/etc/onix-forge.env` in the image). If the pinned ref is a bare commit the
   remote won't advertise, it fetches it explicitly, then detaches to it.

   > **Sidebar — why pin the commit.** moss and boulder are alpha software under
   > active development. Pinning `os-tools` to an exact commit hash means the
   > forge builds the *same* tooling every time, so a green Phase 0 today stays
   > green tomorrow regardless of upstream churn. Rebasing to a newer snapshot
   > is a deliberate act (override `OS_TOOLS_REF`), never accidental.

3. **Checks subuid/subgid** for the build user — the ranges boulder needs for
   its rootless user-namespace sandbox (set up in Phase 002). It warns if
   missing, since rootless builds would then fail.
4. **Builds moss + boulder.** It prefers the project's own bootstrap, `just
   get-started`; if that fails it falls back to `cargo build --release -p moss
   -p boulder` and copies the binaries into `~/.local/bin`.
5. **Verifies** the binaries are on `PATH` and reports their versions.

Result inside the guest:

```text
/home/mason/.local/bin/moss
/home/mason/.local/bin/boulder
```

(`~/.local/bin` was added to `mason`'s `PATH` in `.profile` during Phase 002, so
later steps find them.) The script is idempotent-ish: re-running it updates the
checkout and rebuilds.

## Why it runs over SSH, not in the chroot

boulder and moss are compiled *inside the running VM*, not in the Phase 002
chroot, for two reasons. First, this is a real Rust build that wants cores and
RAM — the VM is sized for it (6 vCPU, 6 GiB by default). Second, and more
importantly, boulder's sandbox uses user namespaces, which need a live,
booted kernel with the right namespace support — not a chroot sharing the host
kernel. The forge has to actually be *running* to build and exercise the tools.

## Conceptually

- `moss` manages installed package states atomically
- `boulder` builds `.stone` packages from `stone.yaml`

"Atomically" and "states" are the words that matter. Every install or removal
moss performs becomes a numbered, immutable state you can return to. That is the
whole reason ONIX uses moss, and it is precisely what [006](./006.md) puts to
the test.

## What it proves — and what it does not

**Proves:** AerynOS's Rust tooling compiles and runs on a musl host. This is the
core feasibility claim behind the entire project — a musl machine plane needs
musl-built tooling, and here it is.

**Does not prove:** that the tools can build or install a *package* yet. That is
[005](./005.md) (cut a `.stone`) and [006](./006.md) (real moss states).

## Gotchas

- **Build takes minutes.** A cold `os-tools` build compiles a lot of Rust; the
  VM is sized for it but be patient on first run.
- **`just get-started` vs cargo fallback.** If upstream's bootstrap recipe
  changes and breaks, the script quietly falls back to a plain `cargo build`, so
  you still get working binaries.
- **`moss not on PATH yet`.** If the final version check prints this, the
  binaries built but the login shell hasn't re-read `.profile`; later phases set
  `PATH="$HOME/.local/bin:$PATH"` explicitly, so they still work.
