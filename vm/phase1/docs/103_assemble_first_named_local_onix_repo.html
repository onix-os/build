Phase 101 and 102 prove individual stones work. Phase 103 proves the next layer: those stones can live in a named repository.

Background: what a moss repo and index are

Up to now the stones have been loose files. You install a loose stone by pointing moss straight at the file path. That does not scale: it means every consumer has to know the exact filename (which includes a version and a build hash) of every package. A repository removes that coupling.

A moss repository, at this scale, is astonishingly simple — it is a directory of .stone files plus one index file:

repo/
  branding-0.1.0-1-....stone
  filesystem-0.1.0-1-....stone
  stone.index          <- generated by `moss index`

moss index <dir> walks the directory, reads each .stone’s metadata (name, version, dependencies, content hashes), and writes stone.index — a manifest that maps package names to the files that provide them. Once a directory has an index, moss can be told “add this as a repo,” and from then on you ask for packages by name and moss figures out which file to fetch. The index is the difference between “a folder with some archives in it” and “a package repository.”

What this step does

It collects:

~/stone-lab/branding/out/*.stone
~/stone-lab/filesystem/out/*.stone

then creates:

~/stone-lab/onix-repo/repo/stone.index

and adds that index to a disposable Moss root as repo name onix-local.

Walking the mechanism, the script (build-local-repo.sh) does the following inside the forge:

  1. Locates the two stones built by steps 101 and 102 in their out/ directories, and errors out with a pointer to make phase 101 / make phase 102 if either is missing. The repo is only ever as current as the last build.
  2. Copies both stones into a fresh repo/ directory and runs moss index repo, then asserts repo/stone.index was created.
  3. Registers the repo under the name onix-local against a throwaway moss root, using a file:// URL that points at the index: 
    moss -D <root> --cache <cache> repo add onix-local "file://<repo>/stone.index" -c "local ONIX phase1 repo"
moss -D <root> --cache <cache> repo update
    -D <root> tells moss to treat a scratch directory as the system root so nothing on the forge is touched; repo update pulls (here, just reads) the index so moss knows what the repo offers.

The file:// local repo

The URL is file://.../stone.index, not https://.... A moss repo does not require a web server — it only needs a fetchable index and fetchable stones. A file:// URL means “the repo is a local directory,” which is exactly the right starting point: it is the same repo mechanism a real HTTPS mirror uses, with the transport swapped for the local filesystem. Step 104 keeps this file:// transport but reshapes the directory into the layout a static web host would serve; a much later phase swaps file:// for https:// with no change to how moss consumes it.

The important proof: install by name

The important proof is that the install happens by package name:

moss ... install --to <target> branding filesystem

Contrast this with steps 101 and 102, which handed moss a specific .stone file. Here moss is given only the names branding and filesystem; it consults the onix-local index, resolves each name to the providing stone, fetches it from the file:// repo, and installs it into the disposable target root. The script then re-asserts the familiar evidence — the generated os-release (NAME="ONIX", ID="onix", the full PRETTY_NAME), the fstab label, and the profile.d scripts — proving a by-name install from a repo yields exactly what the direct-file installs did.

So this is the bridge from “we have loose package files” to “ONIX has the beginning of a package repository.”

Why this matters for the phase gate

The Phase 1 gate is “the onix repo carries a self-consistent base stone set, and moss installs it into a fresh root.” Step 103 is the first time all the words in that sentence are literally true at once: there is a repo (named, indexed), it carries a set (two composing stones), and moss installs it by name into a fresh root. Everything after this — 104 through 108 — is about reshaping and hardening this repo into something you could publish, not about proving the core install works. That core is proven here.

What it proves vs what it does not

It proves: a moss index can be built from ONIX stones, registered as a named repo, and used to resolve-and-install packages by name into a clean root.

It does not prove: anything about a publishable on-disk layout (the repo here is a flat repo/ dir, not the channel/arch tree a web host needs), nor anything about checksums, host export, or a publication plan. Those are steps 104–108. It also still runs entirely inside the forge over a file:// URL — no network, no host artifact yet.
