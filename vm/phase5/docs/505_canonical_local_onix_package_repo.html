ONIX Book

505 — canonical local ONIX package repo

Run:

make phase 505

Phase 505 assembles one local ONIX package repository from the current canonical essential package set.

This is still local.

It does not upload to a server.

It prepares the shape that a later publishing phase can put behind:

https://repo.onix-os.com/unstable/x86_64/stone.index

Why this phase exists

Before Phase 505, ONIX had two useful but separate artifact locations:

artifacts/onix-publish/unstable/x86_64/
artifacts/onix-local-repo/

They came from different historical needs.

The first was built in Phase 1 to prove a publish-style repository layout for:

branding
filesystem

The second was built in Phase 4 to prove booted-base system packages for:

busybox
dropbear
systemd
bootstrap

That split was fine while ONIX was learning.

It is not a good final package plane.

For a real package repository, ONIX needs one canonical repository that can be given to Moss as a single source.

Phase 505 creates that.

The repository shape

Phase 505 writes:

artifacts/onix-repo/
  README.txt
  repo.json
  unstable/
    x86_64/
      MANIFEST.tsv
      SHA256SUMS
      stone.index
      branding-*.stone
      filesystem-*.stone
      busybox-*.stone
      dropbear-*.stone
      systemd-*.stone
      bootstrap-*.stone

The important file for Moss is:

artifacts/onix-repo/unstable/x86_64/stone.index

That is the local equivalent of the future public URL:

https://repo.onix-os.com/unstable/x86_64/stone.index

What is a package repository?

A package repository is not magic.

At this stage, it is just:

a directory full of .stone files
plus an index file that tells Moss what is inside

The .stone files contain package payloads.

The stone.index file is generated by:

moss index path/to/repo

Moss reads that index, learns which packages exist, and can then install by package name.

So instead of saying:

install this exact .stone file

we can say:

install systemd from the onix repo

That is the move from package files toward a distribution package repository.

What moss index actually produces

moss index <dir> scans a directory of .stone files and writes a single stone.index. Think of the index as a table of contents: for each stone it records the package name, version, release, the files/paths that stone owns, its dependencies, and the content hash needed to fetch and verify it. That table is what turns “a folder of files” into “a repository you can query by name.”

The payoff is indirection. A client (moss on the target) never has to know the exact filename systemd-<version>-<release>.stone; it asks the index for systemd and the index answers with the right object. That indirection is the seed of everything a real distro repo does later — channels, upgrades, dependency resolution — but at this stage it is just: name in, stone out.

Atomic assembly: build in temp, then swap

The assembler does not edit artifacts/onix-repo/ in place. It builds the entire tree — copied stones, generated stone.index, MANIFEST.tsv, SHA256SUMS, metadata — inside a hidden temporary directory, verifies it, and only then removes the old repo and mvs the new one into place. This is the same atomic-swap discipline moss uses on /usr: a consumer never sees a half-built repository, because the repo becomes valid in a single rename. If assembly fails partway, the old repo is untouched and the temp tree is cleaned up.

Why this is not public hosting yet

Public hosting adds more questions:

How are files uploaded?
Is the repository signed?
How are old packages retained?
How are bad packages removed?
How do stable and unstable channels differ?
What does the CDN/cache story look like?

Phase 505 avoids those questions on purpose.

It only proves the local repository artifact.

The rule is:

make the local repo boring first, publish later

Source inputs

Phase 505 currently copies stones from the existing artifact roots:

artifacts/onix-publish/unstable/x86_64/
artifacts/onix-local-repo/

It does not rebuild the packages.

Rebuilding belongs to Phase 504.

Phase 505 consumes the already-built stones and assembles one combined repo.

Package set

The first canonical local repo contains:

branding
filesystem
busybox
dropbear
systemd
bootstrap

These are the minimum current ONIX base packages:

  • identity and login branding,
  • filesystem/default policy,
  • bootstrap shell/tool base,
  • SSH daemon,
  • native systemd,
  • bootstrap service policy.

This is not the final full package set.

Rust-first packages such as uutils-coreutils and the selected RootAsRole privilege path should come after the repository lane is stable.

What MANIFEST.tsv is for

The repository index is for Moss.

Humans still need to know where each package came from.

So Phase 505 writes:

MANIFEST.tsv

It maps:

package -> stone filename -> source artifact repo -> PACKAGE.md contract

Example shape:

package                 stone                                source_repo
branding           branding-...stone               artifacts/onix-publish/unstable/x86_64
systemd            systemd-...stone                artifacts/onix-local-repo

This makes the repo auditable.

We can see not only that a stone is present, but also which canonical package contract describes it.

What SHA256SUMS is for

Phase 505 also writes:

SHA256SUMS

It covers:

*.stone
stone.index
MANIFEST.tsv

This is not a final signing system.

It is a local integrity check.

It catches simple corruption or accidental file replacement while we are still building the package plane.

Later repository signing can build on top of this.

Integrity is not authenticity

Keep two ideas separate. SHA256SUMS proves integrity: the bytes you have are the bytes that were written — a truncated download or a silently corrupted stone is caught. It does not prove authenticity: it says nothing about who produced the files, and anyone who can rewrite a stone can also rewrite its checksum line. That is the job of cryptographic signing (a private key the publisher holds, a public key the client trusts), which Phase 5 explicitly defers. The reason to do checksums now anyway is that they catch the boring, common failures — a bad copy, a half-written file, a mismatched pair — for free, and signing later simply layers a signature over the same manifest. Integrity first, authenticity when there is a server worth authenticating.

What the script checks

The helper is:

vm/phase5/assemble-canonical-local-repo.sh

When make phase 505 runs, it checks:

  • host Moss exists,
  • both source artifact repos exist,
  • each required package has exactly one matching .stone,
  • each required package has a canonical PACKAGE.md contract,
  • each input .stone passes moss inspect --check,
  • the combined repo has exactly the expected package set,
  • generated metadata does not use forbidden mixed-case branding,
  • checksums verify,
  • each combined repo .stone passes moss inspect --check.

Then it proves the repo works by installing all essential packages into a scratch target using only:

file://artifacts/onix-repo/unstable/x86_64/stone.index

Install proof

The proof target is:

artifacts/onix-repo-work/install-target/

The script asks Moss to install:

branding
filesystem
busybox
dropbear
systemd
bootstrap

Then it checks for key installed files:

/usr/lib/os-release
/usr/share/onix/filesystem-layout.md
/usr/bin/busybox
/usr/sbin/dropbear
/usr/lib/systemd/systemd
/usr/share/onix/packages/bootstrap.md

This proves the combined repo is not only a directory of files.

It is a real Moss-consumable package source.

Why an install proof, and what file:// means here

A repository that indexes cleanly can still be unusable — a stone might be corrupt, a dependency unsatisfiable, two packages might fight over a path. The only way to know the repo works is to make moss use it exactly as a real client would. So the script spins up a throwaway moss root, adds the repo, and installs the full essential set:

moss -D <work>/moss-root --cache <work>/moss-cache \
  repo add onix-canonical-local file://.../artifacts/onix-repo/unstable/x86_64/stone.index
moss ... install --to <work>/install-target branding filesystem ...

file:// is the key trick. moss speaks URLs; it does not care whether a repository lives behind https://repo.onix-os.com or on the local disk. A file:// URL points moss at the on-disk index using the same code path a network fetch would use. So the install proof exercises the real repository machinery with zero network, zero server, zero DNS — proving the shape is sound before any of that exists. The dedicated -D root and --cache keep the proof hermetic: it installs into a scratch target and touches nothing on the build host’s own system.

Ownership warning discovered here

Phase 505 discovered that old bootstrap artifacts could report a package path collision during the install proof:

/usr/bin/reboot
/usr/bin/poweroff

Early in Phase 5 this happened because busybox provided BusyBox applet links while systemd also provided system-management commands.

Phase 505 did not hide this.

It recorded the warning and still proved the repository can be indexed and installed.

Concretely, moss emits a duplicate entry: line when two packages in one transaction claim the same path. The assembler runs in a lenient ownership mode here (ONIX_REPO_STRICT_OWNERSHIP=0): it notices the collision, prints it, tells you a later phase must remove it, and lets the install proof continue. This is a deliberate sequencing choice — Phase 505 owns “does the repo assemble and install at all?”, and answering that fully is more valuable than aborting on the first known wart. Phase 506 fixed the BusyBox/systemd overlap. Later stones can still reveal new ownership cleanup work, such as the musl loader path, and those are handled as separate package phases instead of being hidden.

Phase 506 fixes the overlap by making ownership explicit:

systemd owns reboot and poweroff
busybox keeps only the bootstrap applet links ONIX wants from it

Why Phase 505 still uses existing artifacts

You might ask:

Why not build every stone directly into artifacts/onix-repo?

That is a good future cleanup.

But Phase 505 is deliberately one smaller step:

504 — prove canonical package builders
505 — assemble a canonical local repo from their outputs

Combining those two ideas in one step would make debugging harder.

If the repo is wrong, Phase 505 owns it.

If a package build is wrong, Phase 504 owns it.

That separation is useful while the system is young.

What comes next

The next step should be:

506 — fix essential package ownership collisions

That follow-up exists because Phase 505 found the reboot/poweroff overlap between busybox and systemd.

After that fix, the plan returns to image consumption:

Right now, older image phases still know about older artifact roots.

Phase 507 should point image assembly at:

artifacts/onix-repo/unstable/x86_64/stone.index

That will prove the image can be assembled from the same repo layout we intend to publish later.
