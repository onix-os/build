This builds a deliberately tiny package called onix-hello.

It creates a local source archive in the guest, writes a stone.yaml, builds a .stone, checks it, extracts it, indexes it into a local repo, installs it into a throwaway target root, and runs:

/usr/bin/onix-hello

Expected output:

hello from onix forge

Why a deliberately boring package

The package does nothing interesting on purpose: /usr/bin/onix-hello is a one-line shell script that prints a greeting. The point is the pipeline, not the payload. If a trivial package can travel the whole road — recipe → build → integrity check → extract → local repo → install → run — then the road itself works, and Phase 1 can start authoring real musl recipes on top of it.

Sidebar — what a .stone is, concretely. A .stone is moss’s package file: a content-addressed archive of the files a package installs, plus metadata (name, version, dependencies, and the file layout). boulder produces them from a stone.yaml recipe; moss consumes them. Unlike a tarball, a .stone records where each file goes and carries hashes so moss can verify integrity and deduplicate content across packages and states.

The recipe

build-hello-stone.sh runs entirely inside the forge (over SSH), writing only under ~/stone-lab/onix-hello. It first creates a source archive (onix-hello-0.1.0.tar.gz) containing the one-line script, hashes it, then writes a stone.yaml that points at that archive by file:// URL and SHA-256:

name        : onix-hello
version     : 0.1.0
release     : 1
summary     : Tiny hello package for proving the ONIX forge
...
upstreams   :
    - file://…/onix-hello-0.1.0.tar.gz: <sha256>
install     : |
    install -Dm00755 onix-hello %(installroot)%(bindir)/onix-hello
    ...

Sidebar — anatomy of a stone.yaml. The recipe declares identity (name, version, release), an upstreams list of source archives keyed by their SHA-256 (so boulder verifies what it downloads), and build phases. The install phase runs after the build and populates a staging tree at %(installroot); boulder macros like %(bindir) expand to standard paths (/usr/bin). Whatever lands under %(installroot) becomes the package’s file layout. Here there is no compile step — the “source” is already a script — so only install matters.

Walk the pipeline

The script drives boulder and moss through the full flow:

  1. Build. boulder build -y --normal-priority -o "$OUT" stone.yaml fetches the (local) source, runs the install phase in boulder’s sandbox, and writes a .stone into $OUT.

  2. Integrity check. moss inspect --check "$STONE" verifies the archive’s internal hashes — the package isn’t corrupt.

  3. Inspect layout. moss inspect "$STONE" prints the package metadata and the file layout moss recorded, so you can see exactly what it will install.

  4. Extract and run (no install). moss extract -o "$EXTRACT" "$STONE" unpacks the payload to a directory, and the script runs …/usr/bin/onix-hello straight out of it — proving the built binary works before any repo or install is involved.

  5. Index a local repo. The .stone is copied into a repo/ dir and moss index "$REPO" generates a stone.index.

    Sidebar — a moss repo is just files. A moss “repository” at this scale is nothing more than a directory of .stones plus a generated stone.index describing them, reachable over file:// (later, static HTTPS). moss index builds that index; moss repo add points a moss root at it. There is no server, no database — which is exactly why ONIX can host its only repo as dumb files. This mirrors how the real onix repo will work in Phase 1.

  6. Install into a throwaway target. The script adds the local repo to a disposable moss root, updates it, and runs moss … install --to "$TARGET" onix-hello, then runs the installed binary at $TARGET/usr/bin/onix-hello.

This proves the package build pipeline works:

source tarball -> stone.yaml -> boulder -> .stone -> moss inspect/extract/install

install --to is not yet a real state

Note that step 6 uses moss … install --to "$TARGET", which blits the files into a target directory. That is enough to prove the package installs and runs, but it is not a real moss state transaction with rollback history — that distinction is the whole subject of 006, which uses moss -D <root> install instead. Keep the two apart: 005 proves building and installing files; 006 proves transactional state.

The recipe gotcha (read this — you will hit it)

Important recipe gotcha learned here:

install     : |
    install -Dm00755 onix-hello %(installroot)%(bindir)/onix-hello
    chmod g-s %(installroot)/usr %(installroot)%(bindir)

Boulder build directories inherit g+s. If /usr keeps that bit, Boulder can emit a /usr/ layout entry that Moss rejects during extract/install.

Unpacking why:

Sidebar — the setgid (g+s) directory bit. When a directory has the setgid bit set, new files and subdirectories created inside it inherit the directory’s group rather than the creator’s. boulder’s build directories carry g+s, so when the install step does install -D …/usr/bin/onix-hello and the parent /usr and /usr/bin get created, they inherit that setgid bit. boulder then records /usr as a special directory in the package layout — and moss currently rejects that special /usr/ entry during extract/install.

The fix is one line: chmod g-s %(installroot)/usr %(installroot)%(bindir) clears the setgid bit off the two directories before boulder captures the layout, so /usr is recorded as an ordinary directory and moss accepts the package. Any simple recipe that installs straight into /usr/bin needs the same chmod g-s. This is documented in ONIX.md as the canonical Phase 0 recipe gotcha, and it is worth internalizing now because every early musl recipe will touch /usr.

What it proves — and what it does not

Proves: boulder builds a valid .stone on musl, and moss can verify, inspect, extract, index, and install it — the complete build-and-install pipeline end to end.

Does not prove: atomic state management or rollback. install --to is a file blit, not a transaction. That final capability is 006, the Phase 0 gate.
