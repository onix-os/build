Why this is the Phase 0 gate

Everything before this proved we can build and install a package. This step proves the one capability ONIX actually exists for: atomic, reversible system state. If moss can install a package as a numbered state, remove it as a second state, and then return to the first state and have the package be present again — cleanly, on musl — then the foundational promise of the whole distribution holds. That is why 006 is the final Phase 0 gate.

Sidebar — what “moss state” means. moss does not just copy files; it records every change to a root as an immutable, numbered state (a transaction). Installing a package creates State #1; removing it creates State #2; and so on. Each state is a complete, self-consistent snapshot of what is installed. Activating an earlier state atomically swaps the system back to that snapshot — a rollback. Because states are immutable and content is deduplicated in moss’s store, rollback is cheap and cannot leave the system half-changed. This is the same mechanism that, in the real ONIX image, will let you break the machine with a bad update and boot back into the previous state.

Why 005 wasn’t enough

Phase 005 uses moss install --to, which blits files into a target directory but does not create a real Moss state transaction.

install --to answers “do the files install and run?” — but it writes them into a plain directory with no transaction history. There is nothing to roll back to, because moss never recorded a state. To exercise the state machine you must install into a real moss root.

Phase 006 uses:

moss -D <root> install onix-hello

That creates real state history under a disposable root.

Sidebar — the -D <root> flag. -D (or --directory) points moss at a root directory to manage, complete with its own .moss content store and transaction history. Unlike install --to, this is moss operating as a real package manager over that root — it writes states into .moss, so state list, state active, and state activate all work. state-smoke.sh uses a throwaway root under ~/stone-lab/onix-hello/state-root so the test is disposable and repeatable.

Walk the smoke test

state-smoke.sh runs inside the forge over SSH and reuses the .stone built in 005 (it fails early with a hint to run make phase 005 if none exists). Then it:

  1. Prepares a disposable moss root, cache, and a local repo directory; copies the .stone in; moss indexes it; and moss … repo add + repo update registers it. A search onix-hello confirms the package is visible.
  2. Records the empty starting state (state list before install).
  3. Installs as a real state: moss -D "$ROOT" … -y install onix-hello. Afterwards it asserts, from the tool’s own output, that the active state is State #1 - Install, that list installed shows onix-hello, and that $ROOT/usr/bin/onix-hello actually runs.
  4. Removes as a second state: moss -D "$ROOT" … -y remove onix-hello. It asserts the active state is now State #2 - Remove, that list installed reports No packages found, and that the binary is gone from the root.
  5. Rolls back by activating State #1: moss -D "$ROOT" … -y state activate 1 --skip-triggers --skip-boot. It asserts the active state is State #1 - Install again, that onix-hello is listed as installed once more, and that the binary runs again.

Each assertion uses grep -q against the captured output, so the script fails loudly if any transition does not produce exactly the expected state — this is a real test, not a demo.

Expected proof:

install onix-hello -> State #1
remove onix-hello  -> State #2
activate State #1     -> rollback works

This is the final Phase 0 gate because atomic state management is the whole reason we are using Moss.

Note — --skip-triggers --skip-boot. In the forge these are safe to skip: we are testing state transitions in a disposable root, not running install triggers or updating a bootloader. In the real ONIX image, activating a state is exactly what promotes a new boot entry and runs triggers — but Phase 0 only needs to prove the state machine itself is sound.

What it proves — and what it does not

Proves: moss creates real, numbered states on install and remove, and can activate an earlier state to bring a removed package back — install → State #1, remove → State #2, activate 1 → rollback, all on musl. This is the capability the entire ONIX design rests on.

Does not prove: anything about booting an ONIX image, systemd, BLS boot entries, or the two-plane (moss + Nix) composition. Those are Phase 1+ concerns. Phase 0 deliberately proves only the tooling and the state machine, in a forge that is then thrown away.

When Phase 0 is complete

Phase 0 is complete when all of these have worked:

  • make doctor
  • make phase 002
  • make phase 003
  • make phase 004
  • make phase 005
  • make phase 006

At that point we know:

  1. we can boot a musl forge,
  2. we can build the AerynOS tools on musl,
  3. we can build a valid .stone,
  4. we can create a local Moss repo,
  5. we can install/remove packages with real Moss states,
  6. we can roll back to an older state.

That maps one-to-one onto the Phase 0 gate from ONIX.md: moss + boulder run on musl; you can boulder-build a hello-world .stone, moss-install it, roll the moss state back, and remove it — cleanly. Six green steps, one proven capability.

Only after this does Phase 1 make sense: real ONIX recipe/repo bootstrap.

What comes next

Phase 1 leaves the toy onix-hello behind and authors the first real musl recipes — the core userland (musl, a toolchain, busybox/coreutils, the essentials onix-base needs) — and stands up the actual onix moss repo. The forge you built here is the workshop where all of that gets cut. Once Phase 1’s base stone set installs into a fresh root that chroots and runs a shell, the forge has done its job and ONIX proper begins.
