422 — build, install, and boot-prove native systemd

Phase 422 is the point where systemd stops being a bootstrap wrapper around a prebuilt runtime payload and becomes a native ONIX package.

This is a big step, but it is still one step on purpose:

make phase 422

That command does three things:

  1. builds systemd from source inside the musl forge VM,
  2. packages the result as a .stone named systemd,
  3. installs it into the ONIX image and boots the image to prove PID 1 still works.

Why this phase exists

Earlier phases proved that ONIX could boot with systemd as PID 1.

That was important, but the first systemd payload was still bootstrap-shaped:

/usr/lib/systemd/systemd -> old external runtime payload

That was acceptable for learning, because it let us answer the first question:

Can a musl ONIX image reach systemd multi-user mode?

Phase 422 asks the next question:

Can ONIX build and own the systemd userspace package itself?

The desired new shape is:

/usr/lib/systemd/systemd

as a real executable file installed by a .stone.

No active runtime symlink should point into the old bootstrap payload.

What native means here

“Native” does not mean “built on the final ONIX system” yet.

At this stage, native means:

  • the source build happens in our Alpine/musl forge VM,
  • the resulting binary targets musl,
  • boulder packages the result as a .stone,
  • moss can install that .stone into an ONIX root tree,
  • the booted ONIX image runs that installed package as PID 1.

The host development shell may still help us acquire or archive the source. That is not runtime ownership.

Runtime ownership means the booted image uses package-owned files under normal image paths such as:

/usr/lib/systemd/systemd
/usr/bin/systemctl
/usr/lib/systemd/system
/usr/share/onix/packages/systemd.md

Refresher: what PID 1 is

When Linux finishes early kernel boot, it needs to start the first userspace process.

That first userspace process has process ID 1:

PID 1

For ONIX, the boot entry tells the kernel:

init=/usr/lib/systemd/systemd

So the kernel tries to execute:

/usr/lib/systemd/systemd

If that file is missing, not executable, or linked against libraries that cannot be found, boot fails very early.

That is why this phase is careful. We are replacing the most important userspace binary in the image.

Refresher: what the ELF interpreter is

Most Linux executables are ELF files.

A dynamically linked ELF executable does not start alone. It names an interpreter, also called the dynamic linker.

For musl on x86_64, that path is normally:

/lib/ld-musl-x86_64.so.1

You can inspect it with:

readelf -l /usr/lib/systemd/systemd

For Phase 422, the expected interpreter is:

/lib/ld-musl-x86_64.so.1

This is important because it is a normal musl system path. It is not a special old bootstrap-store path.

ONIX uses a merged /usr style layout, so /lib resolves through /usr/lib. That means the musl dependency stone installs the loader under:

/usr/lib/ld-musl-x86_64.so.1

and the booted system can execute:

/lib/ld-musl-x86_64.so.1

through the merged layout.

The systemd stone itself must not own that loader. It declares musl as a runtime dependency; the installed target contains the loader because Moss installed the dependency beside systemd.

Background: meson, ninja, and RUNPATH

systemd is built with Meson, a build-configuration system, and Ninja, the tool that actually runs the compile steps. You configure the build once (meson setup, where -D... options select features), then Ninja compiles it (meson compile). This is different from BusyBox and Dropbear, which used the older ./configure && make flow — another reason systemd is a bigger, more delicate build.

When the native systemd binary is linked, ONIX bakes a RUNPATH into it. RUNPATH is a list of directories, stored inside the ELF file, that the dynamic loader searches for this program’s shared libraries. systemd keeps several private libraries under /usr/lib/systemd, so the build records a RUNPATH of /usr/lib/systemd. The Phase 422 build script verifies this with readelf -d; if the RUNPATH were missing, PID 1 would start but fail to find its own private libraries. RUNPATH is the native, package-owned answer to the same problem the bootstrap stone solved by dumping everything into /nix/store.

Why -Dlibc=musl matters

The manual experiment before this phase found a real systemd build gotcha.

If systemd is configured without the musl libc mode, the build tries to use glibc-only headers such as:

printf.h

That fails on musl.

Phase 422 configures Meson with:

-Dlibc=musl

That enables systemd’s musl compatibility include path and lets the source build finish in the forge.

Why this package is still monolithic

Systemd is not like BusyBox or Dropbear.

BusyBox and Dropbear were small enough that we could build mostly self-contained bootstrap packages.

Systemd has more pieces:

  • PID 1 itself,
  • systemctl,
  • journalctl,
  • udevadm,
  • private systemd libraries,
  • unit files,
  • sysusers and tmpfiles policy,
  • helper commands,
  • musl loader/runtime files.

The clean final OS shape would split many of these into separate stones.

But doing that before proving the native boot would create too many variables. If boot failed, we would not know whether the problem was:

  • the systemd build,
  • the package recipe,
  • a missing helper,
  • a missing library,
  • the unit activation,
  • the bootloader entry,
  • or the image materialization.

So Phase 422 intentionally makes the first native systemd package monolithic.

The rule is:

prove the native boot first, split dependencies later

Why readelf was not enough

During this phase we hit an important Linux packaging lesson.

The normal first check for a dynamic executable is:

readelf -d /usr/lib/systemd/systemd

That shows the direct ELF NEEDED libraries. Those are libraries the dynamic linker knows must exist before the program can start.

But direct ELF dependencies are not the whole story.

Programs can also load libraries at runtime with dynamic loading. In practice, systemd can need support libraries such as:

libmount
libblkid
libkmod
libcap
libacl
libpcre2
libfdisk
libsmartcols

Some of these are not obvious just from the direct NEEDED list for PID 1.

The failed boot looked like this:

Failed to initialize kmod context
Failed to mount API filesystems
Freezing execution

That meant the native systemd binary itself started, but it did not have enough runtime support to finish the early PID 1 setup.

So Phase 422 deliberately bundles some immediate non-musl runtime-library families needed for the first native boot proof. This is not the final dependency model. It is a bootstrap-native milestone:

first prove the native PID 1 boot, then split more libraries into cleaner stones

The important exception is musl itself. Once the canonical musl stone exists, there must be exactly one musl owner:

musl         owns /usr/lib/ld-musl-x86_64.so.1 and libc symlinks
systemd depends on musl

This is why the boot probe matters. A package can pass static checks and still fail as PID 1.

What make phase 422 runs

The Phase 4 Makefile target is:

native-systemd-stone:
	./build-native-systemd-stone.sh
	./materialize-etc.sh --native-systemd-stone
	./native-systemd-probe.sh

So the public phase is one command, but internally it has three parts.

Part 1: build the native stone

The builder is:

vm/phase4/build-native-systemd-stone.sh

It:

  1. reads the source identity prepared by Phase 421,
  2. creates a writable source archive,
  3. ensures the forge VM has the native build dependencies,
  4. builds systemd with Meson and Ninja inside the forge,
  5. installs into a temporary destination tree,
  6. creates a prepared package payload,
  7. builds systemd-*.stone with boulder,
  8. verifies the stone with moss,
  9. refreshes the local Phase 4 repo.

The source build happens inside:

/home/mason/stone-lab/systemd-native

inside the forge VM.

The host artifact lands under:

artifacts/onix-stones/
artifacts/onix-local-repo/

Part 2: install into the image

The image materializer action is:

./materialize-etc.sh --native-systemd-stone

It mounts:

artifacts/onix-image/onix.raw

and then:

  1. installs systemd from the local moss repo into a scratch target,

  2. verifies the scratch target has real native files,

  3. removes the old bootstrap systemd activation paths,

  4. copies the native package payload into the image,

  5. activates package-owned bootstrap units into:

    /usr/lib/systemd/system

  6. prunes the old bootstrap systemd runtime compatibility payload,

  7. writes a proof note:

    /usr/share/onix/bootstrap/native-systemd-stone.txt

The materializer is rerun-safe for this transition. If a previous Phase 422 attempt already removed the old bootstrap systemd payload, the native reinstall does not fail just because that old payload is absent.

Part 3: boot-prove

The boot probe is:

vm/phase4/native-systemd-probe.sh

It boots the image with QEMU and checks both serial and SSH paths.

The serial proof checks:

  • PID 1 is systemd,
  • /usr/lib/systemd/systemd exists,
  • /usr/lib/systemd/systemd is not a symlink,
  • systemctl --version works,
  • bootstrap networking still comes up,
  • bootstrap SSH still comes up,
  • the package/proof notes do not point at the old bootstrap runtime path.

The SSH proof repeats the important checks from inside an authenticated remote session.

What success looks like

At the end, the probe should print a success message similar to:

Phase 422 proved the booted ONIX image can use the native source-built
systemd package as PID 1 while bootstrap networking and authenticated SSH
still work.

The live image should have:

/usr/lib/systemd/systemd
/usr/lib/systemd/system
/usr/lib/systemd/user
/usr/bin/systemctl
/usr/bin/journalctl
/usr/bin/udevadm
/usr/share/onix/packages/systemd.md
/usr/share/onix/bootstrap/native-systemd-stone.txt

The old bootstrap systemd compatibility payload should be gone from the active image.

Leaving a native ONIX VM running

The normal proof target is temporary. It boots QEMU, collects evidence, and then stops the VM.

For interactive inspection, use Phase 424:

make phase 424

or the shortcut:

make up

Phase 424:

  1. installs the existing native systemd stone into the image,
  2. runs the same serial and SSH proof,
  3. leaves QEMU running after success.

The SSH endpoint is:

ssh -i vm/state/id_ed25519 -p 7630 onix@127.0.0.1

Inside that VM, a quick sanity check is:

cat /proc/1/comm
systemctl --version

Expected result:

systemd
systemd 259 (259.3)

Stop the running VM with:

make stop

What this does not solve yet

Phase 422 does not mean the whole base OS is finished.

Known remaining work:

  • split bundled helper commands into cleaner dependency stones,
  • split remaining non-musl native runtime libraries into cleaner dependency stones,
  • replace temporary bootstrap unit activation with a real package trigger or preset policy,
  • continue kernel/initramfs/module ownership in Phase 3,
  • revisit logging, getty/login, networking, and normal user experience.

The important win is narrower:

ONIX can build, package, install, and boot its own native musl systemd payload.

That is a major base-system milestone.
