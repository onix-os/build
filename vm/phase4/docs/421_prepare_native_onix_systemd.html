Main proofONIX has a concrete, compressed contract for replacing the Nix-wrapped systemd with a source-built native stone in Phase 422.

Why this phase exists

We made an important decision:

system packages = .stone
Nix = user/toolbox layer

That means systemd must not stay as:

a Nix-built payload wrapped inside a stone forever

The current package is useful, but it is transitional.

Current truth:

systemd.stone exists

But:

the systemd bytes were built by pinned nixpkgs
the systemd runtime still depends on /nix/store paths

So ownership is only half solved.

Phase 421 prepares the next move:

Phase 422 will build and boot-prove a source-built native systemd stone.

Why this is a prep phase, not the build phase

systemd is much larger than BusyBox or Dropbear.

BusyBox could be built as one static binary.

Dropbear could also be built as a small static payload.

systemd is different.

It includes or uses:

  • PID 1,
  • systemctl,
  • journalctl,
  • udevadm,
  • systemd-tmpfiles,
  • systemd-sysusers,
  • udev helpers,
  • unit files,
  • generators,
  • runtime libraries,
  • helper programs from util-linux/kmod/coreutils,
  • compression libraries,
  • musl runtime paths.

So before trying to build it natively, we need to write down exactly what the native build must replace.

That is what Phase 421 does.

The compressed plan

We do not want this:

421 — build musl
422 — build libxcrypt
423 — build acl
424 — build attr
425 — build kmod
426 — build util-linux
427 — build xz
428 — build zstd
429 — build systemd
430 — maybe boot

That would be too fragmented.

Instead, the compressed plan is:

421 — prepare native systemd package contract
422 — build/install/boot-prove native systemd

Internally, Phase 422 may bundle several runtime pieces at first.

That is okay.

The user-facing phase should remain meaningful:

replace wrapped systemd with native systemd

not:

one phase per small library

What “native” means here

In this context, native means:

the systemd runtime is built by ONIX's package flow

For Phase 422, that means:

source build in the forge
package with boulder
install with Moss
boot-prove from the resulting .stone

It also means the installed runtime must not depend on:

/nix/store

The current bootstrap systemd binary has an ELF interpreter like:

/nix/store/...-musl-.../lib/ld-musl-x86_64.so.1

That is the thing we need to remove.

A native ONIX systemd binary should use a normal ONIX musl path. Because ONIX uses merged /usr, the expected interpreter shape is:

/lib/ld-musl-x86_64.so.1

with /lib resolving through the normal ONIX filesystem layout.

Background: the merged-/usr layout

Historically Unix split files across /bin, /lib, /sbin and their /usr counterparts. Modern distributions, ONIX included, use merged /usr: the real files all live under /usr, and the old top-level directories are just symlinks — /lib -> /usr/lib, /bin -> /usr/bin, and so on. This is why a binary can name its loader as /lib/ld-musl-x86_64.so.1 even though the package installs the loader at /usr/lib/ld-musl-x86_64.so.1: the /lib symlink resolves it. Merged /usr also supports ONIX’s atomic model, because it lets the entire OS live in one stateless /usr tree that moss can swap as a unit. The native systemd binary must therefore target the normal musl loader path, so that it resolves through this layout rather than through a borrowed /nix/store path.

Why bundling dependencies first is acceptable

The final clean distro shape is probably:

musl
libxcrypt
acl
attr
kmod
util-linux
systemd

But building all of that as separate stones before boot-proving native systemd would slow down the learning flow.

Originally that meant the first native systemd could be a broad bootstrap-native bundle. After the Phase 5 musl package exists, the rule is tighter: systemd depends on musl and does not bundle the musl loader/libc family.

That means:

musl         carries /usr/lib/ld-musl-x86_64.so.1 and libc symlinks
systemd carries systemd plus temporary non-musl runtime pieces it needs

Then later we can split the bundle into cleaner dependency stones.

This is the same kind of honesty we used before:

first make it work clearly
then split ownership more finely

The critical difference is that the bundle is still source-built for ONIX.

It is not a Nix runtime closure.

The current wrapper truth

Phase 421 reads:

artifacts/onix-image/systemd-payload.out
artifacts/onix-image/systemd-payload.closure

Those files describe the current wrapped systemd payload.

At the time this page was written, the current payload is:

/nix/store/...-systemd-259.3

The important point is not the exact hash prefix.

The important point is:

systemd 259.3 is currently Nix-built

Phase 421 turns that into a native package target:

systemd v259.3 from source

Source policy

Phase 421 may use Nix as a source/reference tool.

That is allowed because we already do this for BusyBox and Dropbear:

Nix finds the locked source
the forge builds from source
boulder packages the output
runtime does not depend on Nix

For systemd, Phase 421 records source identity like:

owner: systemd
repo : systemd
rev  : v259.3

It also records the locked Nixpkgs systemd expression and patch count when that metadata is available locally.

That does not mean ONIX accepts Nix runtime ownership.

It only means:

use the current pinned working source/build knowledge as research

Closure buckets

The current Nix closure includes paths such as:

systemd
musl
libxcrypt
util-linux

Phase 421 classifies them into compressed buckets:

native-systemd-core
phase422-runtime-bundle
phase422-helper-bundle
phase422-library-bundle
phase422-inspect

This bucket file is not the final dependency graph.

It is a practical Phase 422 build map.

In plain English:

build systemd natively,
bundle whatever immediate runtime pieces are needed,
boot-prove,
split later.

The recipe draft

Phase 421 adds:

vm/phase4/stone-recipes/systemd-native/stone.yaml.in

The package name inside that draft is still:

systemd

That is intentional.

We do not want a permanent second package like:

systemd-native

The final package should replace the bootstrap wrapper.

So the native draft keeps the canonical name:

systemd

but moves the implementation from:

wrapped Nix payload

to:

prepared source-built payload

What Phase 422 must prove

Phase 422 should prove:

  • systemd.stone is built from source,
  • the image installs the native package,
  • /usr/lib/systemd/systemd does not point into the old bootstrap store,
  • no active systemd binary needs /nix/store as its interpreter,
  • PID 1 is still systemd,
  • systemctl --version works,
  • bootstrap networking still works,
  • bootstrap SSH still works,
  • the ownership audit no longer reports Nix-built systemd runtime debt.

That is a big phase, but it is one coherent phase.

It answers one important question:

Can ONIX boot with its own source-built systemd package?

What Phase 421 does not do

Phase 421 does not:

  • build systemd,
  • install a new systemd,
  • mutate the disk image,
  • boot QEMU,
  • remove the current bootstrap systemd,
  • remove the current systemd /nix/store closure.

It prepares the contract for doing that safely in Phase 422.

Mental model

Before Phase 421:

we know the current systemd works,
but it is still Nix-built

After Phase 421:

we know exactly what Phase 422 must replace and prove

That keeps the workflow compressed without becoming vague.
